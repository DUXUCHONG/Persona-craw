## 自进化 Agent：从日常对话中进行强化学习

Persona-craw 研究的核心问题：AI Agent 如何从日常自然语言交互中持续改进？

核心论点：

> 日常用语就是隐式奖励。日常对话就是在线 RL。

当前的 LLM Agent 在部署后是静态的。用户的纠正、偏好和反馈在每次会话结束后就被丢弃。我们提出一个系统，将日常交互作为主要训练信号 —— 把自然语言转化为奖励，把经验蒸馏成技能，通过在线强化学习更新 Agent 策略。

研究分为四个方向。

---

## 1. 技能系统：从 Context 到 Memory 到 Skill

### Agent 如何将原始交互转化为持久、可复用的技能？

用户每天通过自然对话纠正、引导和打磨 Agent 的行为。这些交互作为 **Context** 不断积累 —— 散落在各个会话中的纠正、偏好和反馈的原始流。

但原始 Context 无法扩展。一个每天跟用户对话的 Agent，几个月后就会淹没在历史记录里。把对话当作扁平 Memory 存储会导致 context 爆炸、检索噪声和性能下降。

关键洞察：**Context 和 Memory 不是最终形态 —— Skill 才是。**

```
Context（原始交互）
    |
    v
Memory（结构化记录）
    |
    v
Skill（压缩的、可复用的知识）
```

**Context** 是发生了什么 —— 对话、纠正和反应的完整流。信息丰富但噪声大，且无限增长。

**Memory** 是结构化的中间形态 —— 按任务、领域和结果组织的交互记录。比原始 Context 紧凑，但仍然线性增长。

**Skill** 是最终蒸馏形态 —— 带有触发条件、步骤和反模式的可复用策略。紧凑、可组合、在推理时直接可用。它代表 Agent **学到了什么**，而不是 **经历了什么**。

### 1.1 Context → Memory → Skill 转化

当用户在三个不同项目中都说"一律用 pytest"，这是 Context。系统把它组织成 Memory（三条纠正记录，相同模式，不同上下文）。然后蒸馏成 Skill："Python 项目用 pytest 作为测试框架。"

**研究问题：**

- 如何检测原始 Context 中的重复模式，整合为结构化 Memory 记录？
- 如何判断累积的 Memory 何时准备好进行 Skill 蒸馏 —— 什么样的重复阈值、一致性或置信度触发转化？
- 如何从成功（"太棒了"）和失败（"再也不要那样做"）中提取技能？
- 如何处理完整生命周期：Context 持续到达，Memory 周期性整合，Skill 在模式稳定后蒸馏？
- **记忆整合。** 如何合并、压缩和总结相关 Memory 为统一的 Skill —— 类似人类在睡眠中将情景记忆整合为程序性知识？
- **选择性遗忘。** 当 Memory 已被蒸馏为 Skill，原始 Memory 可以丢弃。如何判断哪些 Memory 可以安全遗忘，哪些应保留以供未来技能优化？
- **预算分配。** 在推理时给定固定的 context 窗口，如何在当前任务、检索到的技能和近期对话历史之间最优分配 token？

### 1.2 通过 RL 进行技能进化

技能不应是静态的。随着 Agent 积累更多经验，技能必须进化。

**研究问题：**

- 如何用强化学习随时间优化技能质量？导致用户认可的技能被强化；被纠正的技能被更新。
- 如何处理技能冲突 —— 新偏好与旧技能矛盾时怎么办？
- 如何废弃不再匹配用户行为的技能 —— 将过时技能重新送入 Context → Memory → Skill 管线进行重新蒸馏？

### 1.3 技能层次与组合

不是所有技能都在同一层级。有些是通用原则，有些是狭窄的任务特定技巧。复杂任务需要组合多个技能。

**研究问题：**

- 如何将技能组织成层次：通用原则、领域策略、任务特定流程？
- 如何为复杂任务组合多个技能 —— 例如，搭建项目时组合"代码风格"技能和"测试策略"技能？
- 如何随着证据在不同上下文中积累，将成功的任务特定技能提升到领域级或通用级？

## 2. 师徒学习

### 强模型如何教会小模型有效工作？

为每次用户交互运行大型前沿模型成本太高。但小模型缺乏独立处理复杂任务的推理能力。师徒范式弥合了这个差距。

**师傅**（强模型）探索困难任务，发现有效的执行路径，记录成功和失败。这些知识被蒸馏后提供给**徒弟**（小型、便宜的模型）用于日常执行。

### 2.1 师傅探索

强模型处理复杂、新颖或模糊的任务。它探索多种方案，评估结果，识别什么有效什么无效。

**研究问题：**

- 师傅应该如何记录探索？原始轨迹？蒸馏技能？标注决策树？什么表示形式最适合传递给徒弟？
- 如何在记录成功策略的同时捕获失败原因 —— 让徒弟不仅学会做什么，还学会避免什么？

### 2.2 知识蒸馏格式

师傅的知识必须为徒弟打包。可能的格式包括：

- **技能** —— 带有触发条件和步骤的结构化策略。紧凑、可复用，但可能丢失细节。
- **轨迹** —— 完整执行路径。细节丰富，但存储和检索成本高。
- **决策标注** —— 关键决策点及其原因。在细节和压缩之间取得平衡。
- **反面案例** —— 明确的失败案例加根因分析。防止徒弟重蹈覆辙。

**研究问题：**

- 哪种蒸馏格式在每个 context token 预算下产生最佳徒弟性能？
- 如何组合格式 —— 例如，常规任务用技能，复杂任务用标注轨迹？

### 2.3 徒弟执行

小模型使用师傅蒸馏的知识处理日常交互。

**研究问题：**

- 徒弟如何决定使用师傅蒸馏的技能还是自己的推理？
- 徒弟应该在什么时候升级到师傅 —— 即识别出自己力不从心？
- 如何在师傅探索新领域时持续更新徒弟的知识库？

## 3. 日常语言 RL

### 如何将日常对话转化为强化学习训练循环？

这是 Persona-craw 的核心技术挑战。用户不会提供结构化奖励。他们通过自然语言表达满意、纠正和偏好。系统必须将这些信号转化为 RL 训练。

### 3.1 日常用语转化为 RL 奖励

用户自然说话：

- *"太棒了。"* —— 正向奖励
- *"不对，不要那样。"* —— 负向奖励
- *"我喜欢简短的回答。"* —— 偏好信号
- *"试试另一个 API。"* —— 纠正信号
- *"又错了。"* —— 重复失败惩罚
- 沉默 / 继续任务 —— 弱正向

**研究问题：**

- 如何可靠地将自然语言表达分类为奖励类别（认可、纠正、偏好、不满、中性）？
- 如何处理歧义 —— "有意思"可能是肯定也可能是敷衍？
- 如何传播延迟奖励 —— 第 8 轮的纠正可能指的是第 2 轮的决策？
- 如何提取语言之外的隐式行为信号：重试模式、会话放弃、输出采纳率？

### 3.2 RL 训练算法

一旦奖励被提取，它们必须驱动策略更新。

**研究问题：**

- 哪些 RL 算法最适合稀疏、噪声、自然语言衍生的奖励？候选包括 GRPO（Group Relative Policy Optimization）、DPO（Direct Preference Optimization）和在线 PPO 变体。
- 如何处理极小的批量大小 —— 日常交互产生的数据远少于合成 RL 环境？
- 在真实用户环境中如何平衡探索（尝试新策略）和利用（坚持用户认可的策略）？
- 当奖励在对话级别给出时，如何进行步骤级别的信用分配？

### 3.3 RL 基础设施

从日常交互中进行在线 RL 需要生产级基础设施。

**研究问题：**

- 如何构建对话到训练的管线，将实时交互转化为 (state, action, reward) 元组而不阻塞响应延迟？
- 如何解耦训练循环和服务循环 —— 让 RL 更新不中断用户体验？
- 如何安全地调度训练：空闲时训练、夜间训练、日历窗口训练？
- 如何在训练更新导致性能下降时实现回归测试和自动回滚？

## 4. 本地+云端混合：小型 LoRA-RL + 大型云端模型

### 如何将本地微调的小模型与云端大模型结合？

全部用前沿云端模型太贵。全部用本地小模型牺牲质量。混合方案两者兼得：

- **本地小模型**（通过用户日常交互的 LoRA-RL 微调）快速、便宜地处理常规任务，为用户个性化。
- **云端大模型**处理超出本地模型能力的复杂任务，同时充当蒸馏的"师傅"。

### 4.1 本地 LoRA-RL 个性化

本地模型使用 LoRA（Low-Rank Adaptation）+ RL 持续在用户交互上微调。

**研究问题：**

- 如何在消费级硬件上高效地从日常对话奖励训练 LoRA 适配器？
- 如何防止灾难性遗忘 —— 确保个性化不会摧毁模型的通用能力？
- 如何管理 LoRA 检查点版本控制和质量下降时的回滚？

### 4.2 路由：何时本地，何时云端？

不是每个请求都需要大模型。不是每个请求小模型都能搞定。

**研究问题：**

- 如何构建任务路由器，决定使用本地模型还是升级到云端模型？
- 什么信号表明本地模型力不从心 —— 不确定性估计、任务复杂度启发式、领域不匹配？
- 如何在保持质量的同时最小化云端 API 成本 —— 最优路由阈值是什么？

### 4.3 云端到本地知识回传

当云端模型处理复杂任务时，其执行应该回传给本地模型的训练。

**研究问题：**

- 如何将云端模型的成功执行蒸馏成本地模型的 LoRA 训练数据？
- 随着时间推移，本地模型能否吸收足够知识来处理之前需要云端升级的任务 —— 即本地模型的能力边界是否在扩展？
- 如何衡量和追踪这种能力扩展？

## 研究路线图

### 阶段 1 —— 技能基础

- 从日常对话中提取技能
- 分层技能库，支持检索和注入
- Context → Skill 转化（记忆压缩）
- 基线评估：仅通过技能积累的改进

### 阶段 2 —— 日常用语奖励管线

- 自然语言奖励分类器
- 隐式行为信号提取
- 奖励融合（语言 + 行为 + 任务结果）
- 延迟对话级反馈的信用分配

### 阶段 3 —— 师徒学习 + RL 训练

- 师傅（强模型）探索和知识记录
- 知识蒸馏：技能、轨迹、失败标注
- 徒弟（小模型）使用蒸馏知识执行
- 从日常用语奖励进行在线 RL 训练（GRPO / DPO / PPO）
- 通过 RL 反馈循环进行技能进化

### 阶段 4 —— 本地+云端混合 + 自主进化

- 消费级硬件上的本地 LoRA-RL 个性化
- 本地与云端模型之间的任务路由
- 云端到本地知识回传管线
- 安全更新调度（夜间 / 空闲 / 日历窗口）
- 持续基准测试和回归检测
- 完全自主的自进化 Agent 系统
